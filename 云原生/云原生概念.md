# 1. 什么是云原生
就是一种**专门面向云环境设计、开发、部署和运维应用的技术理念与方法**。
它不是某一个具体软件，而是一套思想和技术组合。目标是让应用更容易做到：
**快速发布、弹性扩缩容、高可用、易维护、自动化运维。**
传统应用是：

> 先开发一个系统，再想办法部署到服务器上。

云原生应用是：

> 从一开始就按照云环境的特点来设计系统，让它天然适合在云上运行。

也就是说，云原生强调应用要适应云的能力，比如自动扩容、故障恢复、容器部署、服务拆分、自动化运维等。
# 2. 云原生核心技术
常见的云原生技术主要包括：

|核心技术|作用|
|---|---|
|容器|把应用和运行环境打包在一起，常见工具是 Docker|
|容器编排|管理大量容器，常见工具是 Kubernetes|
|微服务|把一个大系统拆成多个小服务|
|DevOps|开发、测试、部署、运维流程自动化|
|CI/CD|自动构建、自动测试、自动发布|
|服务网格|管理服务之间的通信、监控、安全，例如 Istio|
|可观测性|日志、指标、链路追踪，用来发现和定位问题|
# 3. 云原生和传统部署的区别
| 对比项  | 传统应用     | 云原生应用     |
| ---- | -------- | --------- |
| 部署方式 | 手动部署到服务器 | 容器化自动部署   |
| 系统结构 | 单体应用较多   | 微服务较多     |
| 扩容方式 | 人工加服务器   | 自动扩缩容     |
| 故障处理 | 人工排查重启   | 自动恢复、自动迁移 |
| 发布方式 | 周期长、风险高  | 小步快跑、持续发布 |
| 运维方式 | 人工操作多    | 自动化、平台化   |
# 4. 云原生不等于上云
这是很多人容易混淆的地方。
把一个传统系统直接搬到云服务器上，只能叫：

> 上云 / 云部署

不一定叫云原生。
真正的云原生要求系统本身在架构、部署、运维方式上都适合云环境。
例如：

```
传统 Java Web 项目部署到阿里云 ECS
```

这只是上云。
如果它被容器化、拆成微服务、用 Kubernetes 管理、支持自动扩容、自动发布、监控告警，那才更接近云原生。
