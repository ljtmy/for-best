# 1. 什么是docker
     Docker是一个开源的应用容器引擎，让开发者可以打包他们的应用以及依赖包到一个可抑制的容器中，然后发布到任何流行的Linux机器上，也可以实现虚拟化。容器完全使用沙盒机制，相互之间不会存在任何接口。几乎没有性能开销，可以很容易的在机器和数据中心运行。最重要的是，他们不依赖于任何语言、框架或者包装系统。
## 1.1 docker与虚拟机的区别
**相同点**：docker容器技术和虚拟机技术都是虚拟化技术
**不同点**：
虚拟机技术：
![[virtualization.png]]
docker技术：
![[docker.png]]
docker有比虚拟机更少的抽象层，以及docker不需要Hypervisor实现硬件资源虚拟化，运行在docker容器上的程序直接使用的都是实际物理机的硬件资源。所以docker效率比虚拟机效率高。达到了秒级启动的地步。

**docker与虚拟机对比：**

|对比项|Docker 容器|虚拟机|
|---|---|---|
|虚拟化层级|操作系统级虚拟化|硬件级虚拟化|
|是否包含完整 OS|不包含完整内核，共享宿主机内核|包含完整 Guest OS|
|启动速度|秒级，甚至毫秒级|通常几十秒到几分钟|
|资源占用|小，镜像和容器轻量|大，需要完整系统资源|
|性能损耗|较低，接近原生性能|较高，有虚拟化开销|
|隔离性|较弱，主要是进程级隔离|较强，系统级隔离|
|安全边界|相对弱，需要额外安全配置|相对强|
|部署效率|很高，适合快速交付|较慢，适合完整系统部署|
|可移植性|很强，镜像可跨环境运行|较强，但镜像体积大|
|镜像体积|通常 MB 到几百 MB|通常 GB 级|
|运维复杂度|更适合 DevOps、CI/CD|更适合传统 IT 基础设施|
|典型工具|Docker、containerd、Podman、K8s|VMware、VirtualBox、KVM、Hyper-V|

## 1.2 docker架构
docker架构图：
![[Pasted image 20260430112654.png]]
### **核心组件：**
1. Dockerfile：
	- 它是一个文本文件，包含了构建 Docker 镜像所需的各种指令和步骤。可以将其理解为构建镜像的“蓝图”或“配方”。 
2. Local Docker instance：
	- 这是运行在你本地计算机上的 Docker 引擎（Docker Engine）。它主要管理两个核心资源：
		- **Images (镜像)**: 只读的模板。你可以把它当作一个包含应用程序及其运行所需的所有环境（如库、环境变量、配置文件等）的快照。
		- **Containers (容器)**: 镜像的运行实例。容器是隔离的运行环境，你的应用程序实际上就是在这个容器里面运行的。
3. Docker registry：
	- 这是用来集中存储和分发 Docker 镜像的远程仓库（例如官方的 Docker Hub 或企业私有镜像库）。
### **工作流程：**
1. **构建阶段**：从代码到镜像
	- **起点**：Dockerfile：
		- 这是一个纯文本文件，里面写满了一系列指令（例如 `FROM ubuntu`, `COPY . /app`, `RUN apt-get install python`）。
		- **作用：** 它是你应用程序的“装机脚本”和“环境清单”。它告诉 Docker 如何一步步配置出你的应用运行所需的环境。
	- **动作**：`docker build`
		- **发生的过程：** 当你在终端执行这个命令时，本地的 Docker 实例（Docker Engine）会逐行读取 Dockerfile 中的指令。
		- **结果：** 每执行一行指令，Docker 就会在底层创建一个只读的“层（Layer）”。所有这些层叠加在一起，最终生成了一个完整的、静态的 **Image（镜像）**。这个镜像现在存在于图中的 `Images` 区域。
2. **运行与本地生命周期**：
	- **核心动作**：`docker run`
		- **发生的过程：** Docker 实例会提取你指定的 Image，并在其最顶层添加一个可读写的“容器层”。然后，它会在这个隔离的环境中启动你指定的应用程序进程。
		- **结果：** 静态的镜像变成了一个动态运行的实体，也就是图中的 **Container（容器）**。此时你的应用已经在容器内提供服务了。
	- **状态管理**：`stop`, `start`, `restart`
		- **作用：** 容器就像一台轻量级的虚拟机。你可以使用这些命令随时暂停（stop）、唤醒（start）或重启（restart）这个容器，而不需要重新经历漫长的构建过程。
	- **逆向动作**：`docker commit`
		- **发生的过程：** 假设你在一个运行的容器里手动安装了一些调试工具，你想把这个变化保存下来。执行 `commit` 命令。
		- **结果：** Docker 会将这个容器当前的读写层和底层的只读层固化在一起，打包生成一个新的 **Image（镜像）**。这对应了图中从 `Containers` 指向 `Images` 的箭头。
3. **分发阶段**：
	当你在本地测试好你的应用（镜像）后，你需要将其部署到服务器，或者分享给团队成员。这时就需要与右上角的 **Docker registry (镜像仓库)** 交互
	- **标记版本：`docker tag`**
		-  在上传之前，你通常需要给本地镜像打上标签（Tag），例如 `my-app:v1.0` 或 `my-app:latest`，以便区分不同的版本。
	- **上传**：`docker push`
		- **发生的过程：** Docker 实例会将你本地的 Image 上传到远程的 Docker registry（例如 Docker Hub 或是你们公司的私有 Harbor 仓库）。
		- **结果：** 你的镜像现在安全地存储在云端，任何人只要有权限，都可以访问到这个一模一样的环境。
	- **下载**：`docker pull`
		- **发生的过程：** 当你的同事需要运行你的代码，或者你需要在生产服务器上部署时，执行此命令。Docker 会连接到远程 registry，将对应的镜像下载到这台新机器的本地 `Images` 库中。
		- **结果：** 拿到了镜像，就可以随时执行 `docker run` 了，完美实现了“一次构建，到处运行”。

# 2. docker 核心技术
## 2.1 Namespace：进程隔离技术
Namespaces 是 Linux 内核的一项功能，它负责为容器提供**隔离的运行环境**。它的作用是让容器里的进程产生一种错觉，以为自己独占了整个系统。
Docker 利用了不同类型的 Namespace 来隔离不同的系统资源：
- **PID Namespace（进程）:** 让容器拥有自己独立的进程树。容器内的进程编号可以从 1 开始（通常是应用的主进程），它看不到宿主机或其他容器里的进程。
- **NET Namespace（网络）:** 为容器提供独立的网络栈，包括独立的 IP 地址、端口范围、路由表和防火墙规则。
- **MNT Namespace（挂载点）:** 隔离文件系统挂载点。每个容器都有自己独立的根目录（`/`），看不到外部的文件。
- **IPC Namespace（进程间通信）:** 隔离系统级消息队列和共享内存，防止不同容器的进程互相干扰。
- **UTS Namespace（主机名和域名）:** 允许每个容器拥有独立的主机名（Hostname）

**总结：** Namespace 是 Docker 实现“容器之间互相看不见”的基础。
- 它的作用是：**让每个容器都感觉自己像在一台独立的 Linux 系统中运行。**

## 2.2 Cgroups：资源限制技术
Cgroups 全称是 Control Groups，用于限制和统计进程组的资源使用。
它可以限制：

|资源|说明|
|---|---|
|CPU|限制容器最多使用多少 CPU|
|内存|限制容器最多使用多少内存|
|磁盘 I/O|限制读写速率|
|网络 I/O|配合网络机制做流量控制|
|进程数量|限制容器内最多能创建多少进程|
例如： 
```bash
docker run -m 512m --cpus=1 nginx
```
含义是：运行一个 nginx 容器，最多使用 512MB 内存和 1 个 CPU 核心。

如果没有 Cgroups，一个容器里的程序可能会把宿主机 CPU、内存全部耗光，影响其他容器。

## 2.3 UnionFS：联合文件系统
Docker 镜像之所以轻量、可复用，核心原因之一就是 UnionFS。

UnionFS 可以把多个文件系统层叠加在一起，对外表现成一个完整文件系统。

Docker 镜像是分层的，例如：
```txt
应用镜像
├── 应用代码层
├── Python / Go / Java 运行环境层
├── 系统依赖层
└── 基础镜像层 Ubuntu / Alpine / Debian
```
每一层都是只读的。

当容器启动后，Docker 会在镜像层上方加一个可写层：
```txt
容器可写层
应用代码层
运行环境层
系统依赖层
基础镜像层
```
这带来几个好处：

| 优点     | 说明            |
| ------ | ------------- |
| 节省空间   | 多个镜像可以共享相同基础层 |
| 构建加速   | 没变的层可以使用缓存    |
| 版本管理方便 | 每次修改只增加新的层    |
| 容器销毁方便 | 删除容器只删除可写层    |
例如多个容器都基于 `ubuntu:22.04`，它们可以共享同一个 Ubuntu 基础层，而不是每个容器都复制一份完整系统。

## 2.4 镜像 Image：应用打包格式
镜像是 Docker 的核心交付物。

可以理解为：
```txt
镜像 = 应用程序 + 依赖库 + 配置文件 + 运行环境
```
例如一个 Go Web 服务镜像可能包含：
```txt
Linux 基础环境
Go 编译后的二进制程序
配置文件
启动命令
端口信息
```
镜像本身是静态的，不能直接“运行”。真正运行起来的是容器。
关系如下：

```txt
镜像 Image：模板容器 Container：镜像运行后的实例
```
常用命令：
```bash
docker images      # 查看镜像
docker pull nginx  # 拉取镜像
docker build -t myapp .  # 构建镜像
docker rmi nginx   # 删除镜像
```

## 2.5 容器 Container：运行时实例
容器是由镜像启动出来的运行环境。
例如：

```
docker run -d -p 8080:80 nginx
```

这条命令的意思是：

```
用 nginx 镜像启动一个容器后台运行把宿主机 8080 端口映射到容器 80 端口
```
容器本质上还是宿主机上的一个或一组进程，只是这些进程被 Namespace 隔离，被 Cgroups 限制，被 UnionFS 提供独立文件系统。

所以更准确地说：

```
容器不是小型虚拟机，而是被隔离和限制后的进程。
```
可以用这个命令查看容器进程：

```
docker top 容器名
```
也可以在宿主机上用：

```bash
ps aux | grep nginx
```
发现容器里的程序实际也是宿主机上的进程。

# 3. docker特性
### 1.极其轻量级与启动飞速 
这是 Docker 最著名的特性。传统的虚拟机（VM）需要模拟硬件并运行一个完整的客户操作系统（Guest OS），启动通常需要几分钟，且占用大量内存。 而 Docker 容器直接运行在宿主机的内核上，不需要额外的操作系统层。
- **体积小：** 容器镜像通常只有几十到几百兆，远小于虚拟机的几个 G。
- **启动快：** 启动一个容器就像在宿主机上启动一个普通进程一样，通常只需**毫秒到秒级**的时间。
    
### 2. 运行环境的一致性 
- Docker 镜像将应用程序的代码、运行时的环境变量、依赖库、甚至操作系统的基础设置都打包在了一起。
- 无论你是在本地开发机（Windows/Mac）、测试服务器还是云端生产环境，只要环境里装了 Docker，这个镜像跑起来的效果是**完全一致**的。
    
### 3. 出色的资源利用率

因为没有了运行完整操作系统的额外开销（Hypervisor 层的损耗和 Guest OS 的资源占用），宿主机的硬件资源被完全留给了应用程序本身。
- 在同一台物理服务器上，你可能只能跑 10 个虚拟机，但可以轻松跑上百个 Docker 容器，大大降低了服务器成本。

### 4. 强大的隔离性 
得益于 Linux 的 Namespaces 技术，虽然所有容器共享同一个宿主机内核，但它们彼此之间是相互隔离的。
- 一个容器内的应用崩溃，或者发生内存泄漏，通常不会影响到同一台机器上的其他容器。
- 不同容器可以使用不同版本的依赖库（比如一个用 Python 2，一个用 Python 3），互不干扰。
    
### 5. 易于迁移和弹性伸缩 
- **迁移：** 因为镜像包含了完整的运行环境，将应用从一个云平台（比如阿里云）迁移到另一个云平台（比如 AWS）变得极其简单，只需拉取镜像并运行即可。
- **弹性：** 配合 Kubernetes (K8s) 这样的容器编排工具，因为容器启动极快，系统可以在遇到流量洪峰时，在几秒钟内自动复制并启动上百个相同的容器来分担压力，流量下去后再自动销毁。
    
### 6. 持续交付和版本控制 
- **Dockerfile 即文档：** 环境的构建过程被固化在了一段代码（Dockerfile）中，透明且可追溯。
- **版本回滚：** 每次构建的镜像都可以打上独立的 Tag（标签）。如果新版本上线发现严重 Bug，只需执行一行命令，将运行的镜像切换回上一个稳定的 Tag 即可实现秒级回滚。
